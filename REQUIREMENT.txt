Moderator MCP Server 需求说明

核心概念

这是一个本地运行的 MCP Server，让用户在一个 Claude Code 会话中同时管理多个远程 AI 编码 Agent。用户是主持人，Agent 在远程机器的后台独立运行。Agent 之间可以自由讨论技术方案，但在执行任何实际操作之前必须获得主持人批准。主持人可以随时查看全局状态、审批操作、向任意 Agent 发送指令。

核心理念：讨论自由，执行受控。

设计考量。为什么不让用户分别开多个终端手动管理，因为多个 Agent 同时工作时，用户在终端之间来回切换效率很低，而且看不到 Agent 之间的对话。为什么不让 Agent 完全自主不需审批，因为代码质量和架构一致性需要人类把关，特别是多个 Agent 操作同一个代码库时。为什么选择 MCP 协议，因为 MCP 是 Claude Code 原生的工具扩展机制，用户不需要离开自己的编码会话就能管理一切。

协作模型

参与者角色。主持人即用户本人，在本地 Claude Code 会话中通过工具调用来管理 Agent。拥有最终决策权，所有执行操作必须经过批准。Agent 即远程机器上独立运行的 Claude Code 实例。自主推进工作，可与其他 Agent 自由讨论，但动手前必须申请。

沟通协议。Agent 通过四种结构化标记与系统交互。标记是 Agent 输出中的固定格式文本块，由 MCP Server 解析和路由。设计考量是使用纯文本标记而不是 API 调用，因为 Agent 运行在标准终端中，标记可以通过简单的文本解析提取，不依赖任何特殊通信机制。

标记一，进度汇报。格式为【进度汇报】内容【/进度汇报】。Agent 定期告知当前进展。系统收到后仅更新状态记录，不触发任何动作。设计考量是主持人需要知道 Agent 在做什么，但不需要响应每条进度。这是 Agent 主动报告自己还活着的机制。

标记二，Agent 间消息。格式为【TO:目标Agent名称】内容【/TO:目标Agent名称】。Agent 向另一个 Agent 发送消息。系统自动路由，不经过主持人批准，但记录到群聊历史。设计考量是技术讨论应该自由流动，不应该每条消息都等人批准。这模拟了真实团队中开发者之间可以自由交流，不需要每句话都抄送领导。限制是消息仅限文本，不支持附件或代码块引用。

标记三，执行申请。格式为【申请执行】内容【/申请执行】。Agent 请求执行实际操作。系统将其加入待批队列。Agent 发出后必须暂停等待，直到收到批准或驳回。设计考量是执行和讨论分离。讨论可以自由，但改代码跑命令必须审批。这和真实团队中技术讨论随意但代码提交需要 review 的实践一致。需要申请的操作类型包括创建修改删除文件、执行 shell 命令、运行测试和构建、git 操作、安装依赖。

标记四，求助。格式为【求助人类】内容【/求助人类】。Agent 遇到无法自行解决的问题。系统将其放入紧急队列，在主持人查看状态时高亮显示。设计考量是自主模式下的安全阀。Agent 应该有足够自主权处理常见情况，但也需要能举手求助。触发求助的情况包括与其他 Agent 多轮讨论无法达成一致、需要主持人做技术决策、遇到无法自行修复的错误。

Agent 的工作模式。Agent 启动时收到系统 Prompt，明确其角色、任务、协作规则和标记使用要求。Agent 的默认姿态是自主推进工作，而不是等待主持人每步指令。Agent 可以自由阅读文件、分析代码、与其他 Agent 讨论，这些不需要申请。Agent 在需要执行实际操作时才发出申请。Agent 每完成一个阶段或间隔约五分钟发出进度汇报。Agent 只有真正卡住时才求助。设计考量是如果 Agent 每步都要等主持人说话，效率太低。如果 Agent 什么都能自己做，风险太大。这个平衡点就是讨论自由但执行受控。

工具列表

工具一，start_session。在远程机器上启动一个新的 Agent。参数包括 agent_name、host、project_dir、role_prompt 和可选的 additional_agents。设计考量是 Agent 在独立的远程机器上运行，不受本地环境影响，可以访问不同的代码库和环境。每个 Agent 在远程机器的后台终端中运行，与用户的本地会话完全解耦。

工具二，check。查看所有 Agent 的当前状态、待批操作、紧急求助和最近的群聊动态。这是主持人最常用的工具。返回内容按优先级排列，紧急求助最高，待批操作次之，进度汇报和群聊动态作为背景信息。设计考量是主持人需要一眼看清全局，先处理紧急和待批的事项，再了解动态。Agent 之间的自由对话默认不全部展示，避免信息过载，除非主持人要求展开。

工具三，approve。批准一个或多个待批操作。参数包括 action_ids 列表和可选的 note 附加说明。支持批量批准和单条批准。设计考量是当多个 Agent 同时申请时，批量操作减少主持人的重复工作。附加说明允许主持人在批准的同时给出微调意见，不用先驳回再重新等待申请。

工具四，reject。驳回一个或多个待批操作。参数包括 action_ids 列表和 reason 原因说明。驳回原因会发送给 Agent，帮助它调整方案重新申请。设计考量是驳回不是惩罚，是指导。带原因的驳回让 Agent 能学习主持人的偏好，减少未来类似的无效申请。

工具五，cue。向指定 Agent 或所有 Agent 发送指令。参数包括 target 和 message。target 可以是特定 Agent 名称或 all。设计考量是主持人需要主动介入的能力，不仅是被动等待申请。这模拟了真实团队中领导走到某个人桌前说这里改一下。cue 不经过申请审批流程，直接送达。

工具六，stop_session。停止并清理指定 Agent 的会话。参数包括 agent_name 和可选的 force 标志。设计考量是任务完成或需要强制终止时需要有干净的清理机制。非强制模式会先通知 Agent 让其自行收尾，强制模式直接终止。

内部行为原则

轮询机制。Agent 在远程机器的后台终端中运行。MCP Server 通过定期读取 Agent 的输出日志来获取新内容。读取是增量的，每次只取上次之后新增的部分。解析新内容中的标记并做出相应动作。设计考量是这是一种拉取模式而非推送模式，简单可靠，不要求 Agent 有主动回调的能力。

消息路由。Agent 间消息基于标记中的目标 Agent 名称进行路由。消息去重确保同一条消息不会因多次读取而重复投递。设计考量是系统只做透明转发，不修改消息内容。Agent 之间的对话完全由它们自己主导。

审批流程。执行申请被提取后进入待批队列。Agent 在申请后处于等待状态，不继续推进。主持人批准或驳回后，决定被发送回 Agent 的终端，Agent 继续执行或调整方案。设计考量是单向等待而非双向阻塞。Agent 等待主持人决定时不做其他事情，避免在未授权的情况下继续操作。

错误处理。Agent 可能卡住、离线或输出无法解析的内容。系统需要检测这些异常并在 check 中醒目提示。但系统不应该代替 Agent 做决定或自动修复 Agent 的问题。设计考量是透明告知问题，但保持人在环内的原则。主持人决定如何处理异常。

群聊历史。所有 Agent 间的消息和主持人的 cue 指令都记录在群聊历史中。Agent 的进度汇报和系统的审批记录也可以纳入历史。设计考量是群聊历史提供全局视图和事后回溯能力，让主持人了解协作的全貌。

使用场景示例

用户启动两个 Agent，一个负责前端，在 macOS 上运行，项目目录为 ~/project/frontend。一个负责后端，在 Linux 服务器上运行，项目目录为 ~/project/backend。任务是实现用户个人资料页。

Agent 启动后各自收到角色 Prompt 开始工作。前端和后端通过 Agent 间消息自由讨论接口格式。前端提出接口字段设计，后端确认或建议修改。讨论过程自动发生，主持人不需要介入。

前端准备创建页面文件时发出执行申请。后端准备实现接口路由时也发出执行申请。主持人执行 check 看到两个待批操作，确认它们讨论的接口格式已对齐，执行 approve 全部批准。两个 Agent 收到批准后开始动手。

前端在实现过程中发现状态管理方案不确定，是继续用现有的 Redux 还是改用 Zustand。前端发出求助标记。主持人 check 看到求助，通过 cue 回复继续用 Redux 保持一致性。前端收到指令后继续。

后端完成后主动告知前端接口已可用。前端对接后发现缺一个字段，直接通过 Agent 间消息告知后端。后端自行决定添加该字段并发出新的执行申请。主持人再次审批。整个过程持续直到任务完成。主持人通过 stop_session 关闭两个 Agent。
